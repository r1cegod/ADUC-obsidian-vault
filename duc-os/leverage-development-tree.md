> **TL;DR**: The next 3 months are the foundation-to-leverage development tree: Raven becomes usable, distribution turns proof visible, and automation compresses eval/reporting/repurposing so every build creates learning, proof, distribution, and opportunity signal.

## Growth Contract
- Parent branch: [[duc-os/escape-route]]
- Node role: leaf
- First parent link: [[duc-os/escape-route]]
- Growth trigger: split only when a specific engine such as distribution, paid pain, proof packaging, or weekly dashboard needs its own operating leaf.
- Forbidden contents: raw journaling, vague motivation, full project implementation logs, unverified Singapore legal/admission requirements, and feature wishlists not tied to leverage.
- Source/evidence boundary: this page is a strategic development tree. Project code/eval evidence stays under project branches; current external requirements need separate verified research before execution.

## Core Assumption

```text
usable time: ~4h/day average
real productive time: 18-24h/week
mode: build automation first, then use automation to compound
```

This is not a normal productivity plan. It is a factory game.

```text
build machine
  -> machine reduces next week's cost
  -> saved cost creates more output
  -> output becomes proof
  -> proof creates distribution
  -> distribution creates opportunity signal
```

## Root Tech Tree

```text
ROOT: Escape Velocity System

1. Duc OS
   -> decision / routing / drift prevention

2. Raven
   -> signal filtering / source intelligence / judgment proof

3. Evaluation Automation
   -> cheaper iteration / less manual audit drag

4. Proof Packaging Engine
   -> turns private builds into readable credibility

5. Distribution Engine
   -> turns proof into attention and trust

6. Room / Opportunity Engine
   -> turns attention into better people, paid pain, intros

7. Money / Geography Engine
   -> turns opportunity into optionality and escape routes
```

Compounding chain:

```text
Raven works
  -> faster learning
  -> stronger artifacts
  -> proof packaging
  -> distribution
  -> better rooms
  -> paid/institutional opportunity
  -> escape options
```

## Leverage Law

Every foundational system must affect at least one of these:

```text
learn faster
build faster
prove better
distribute wider
earn sooner
enter better rooms
```

If a system does not hit one of those, it is not foundational leverage. It is complexity.

## Automation Priority

### S-Tier

```text
eval runners
report generation
proof packaging
distribution repurposing
feedback / pain capture
```

These directly compress the compounding loop.

### A-Tier

```text
weekly dashboard
portfolio update helper
source-intelligence templates
```

These are useful once the core loop exists.

### B-Tier

```text
UI polish
social scheduling
fancy analytics
```

These matter later, but they are not first-order escape leverage.

### Trap-Tier

```text
automating before workflow is proven
building dashboards for metrics that do not matter
adding agents to avoid hard judgment
```

The rule:

```text
automate repeated pain, not imagined elegance
```

## Failure Cases

### Failure Case 1 - Private Raven

```text
Raven works privately
  -> no public artifact
  -> no one sees judgment
  -> no external pull
```

Fix:

```text
every strong Raven loop must create a proof artifact
```

### Failure Case 2 - Distribution Without Proof

```text
posting starts before proof is sharp
  -> generic AI content
  -> weak trust
```

Fix:

```text
publish from real build/eval evidence only
```

### Failure Case 3 - Automation Addiction

```text
builds automation for imagined future scale
  -> avoids current hard judgment
```

Fix:

```text
only automate the bottleneck that was hit twice
```

### Failure Case 4 - New Project Escape Fantasy

```text
new project creates fresh dopamine
  -> existing leverage loop remains unfinished
```

Fix:

```text
no new big project before Month 3 route selection
```

## Weekly Review

Run once per week:

```text
1. What machine was upgraded?
2. What cost will be lower next week?
3. What proof was created?
4. What proof was packaged?
5. What got distributed?
6. What feedback or pain signal returned?
7. What machine is now the bottleneck?
```

Score outputs, not emotion.
